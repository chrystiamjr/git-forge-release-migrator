# CLI Reference (English)

This document reflects the current Dart CLI contract (`gfrm`).

## Canonical Commands

```bash
./bin/gfrm migrate [options]
./bin/gfrm resume [options]
./bin/gfrm demo [options]
./bin/gfrm setup [options]
./bin/gfrm settings <action> [options]
```

> In local development, prefix commands with `./bin/gfrm`. When using a compiled release artifact, use `./gfrm` (or `.\gfrm.exe` on Windows) directly. All examples below use `gfrm` without a path prefix for brevity.

## Local Development (Yarn First)

Recommended local checks from repository root:

```bash
yarn lint:dart
yarn test:dart
./scripts/smoke-test.sh
```

Direct Dart commands still work, but docs prioritize `yarn` scripts for day-to-day local development.

## Running CI Artifacts

Artifacts generated by CI:

- `gfrm-macos-intel.zip` containing `gfrm` (Intel Macs)
- `gfrm-macos-silicon.zip` containing `gfrm` (Apple Silicon Macs)
- `gfrm-linux.zip` containing `gfrm`
- `gfrm-windows.zip` containing `gfrm.exe`

macOS (Intel):

```bash
unzip gfrm-macos-intel.zip -d ./gfrm-macos-intel
cd ./gfrm-macos-intel
chmod +x ./gfrm
./gfrm --help
```

macOS (Apple Silicon):

```bash
unzip gfrm-macos-silicon.zip -d ./gfrm-macos-silicon
cd ./gfrm-macos-silicon
chmod +x ./gfrm
./gfrm --help
```

Linux:

```bash
unzip gfrm-linux.zip -d ./gfrm-linux
cd ./gfrm-linux
chmod +x ./gfrm
./gfrm --help
```

Windows (PowerShell):

```powershell
Expand-Archive .\gfrm-windows.zip -DestinationPath .\gfrm-windows
.\gfrm-windows\gfrm.exe --help
```

Notes:

- Release workflow supports permissive/strict macOS security mode (`MACOS_RELEASE_SECURITY_MODE`) for signing/notarization.
- In strict mode, macOS release jobs fail when signing/notarization credentials are missing or notarization fails.
- If Gatekeeper still blocks execution, use troubleshooting fallback: `xattr -d com.apple.quarantine ./gfrm`.
- Windows SmartScreen warnings are expected while binaries are unsigned.

## Providers and Aliases

Supported providers:

- `github` (alias: `gh`)
- `gitlab` (alias: `gl`)
- `bitbucket` (alias: `bb`, Bitbucket Cloud only)

Supported cross-forge pairs:

- `gitlab -> github`
- `github -> gitlab`
- `github -> bitbucket`
- `bitbucket -> github`
- `gitlab -> bitbucket`
- `bitbucket -> gitlab`

Not supported:

- same-provider migrations
- Bitbucket Data Center / Server

## Usage Examples

### Basic migration (GitLab → GitHub)

```bash
gfrm migrate \
  --source-provider gitlab \
  --source-url "https://gitlab.com/group/project" \
  --target-provider github \
  --target-url "https://github.com/org/repo"
```

### Migration with tag range

Migrate only releases between two tags (inclusive):

```bash
gfrm migrate \
  --source-provider gitlab \
  --source-url "https://gitlab.com/group/project" \
  --target-provider github \
  --target-url "https://github.com/org/repo" \
  --from-tag v1.0.0 \
  --to-tag v2.5.0
```

### Dry-run before migrating

Validate source/target access and simulate migration without writing to the target:

```bash
gfrm migrate \
  --source-provider gitlab \
  --source-url "https://gitlab.com/group/project" \
  --target-provider github \
  --target-url "https://github.com/org/repo" \
  --dry-run
```

### Resume an interrupted migration

If a run was interrupted, resume from the last saved session:

```bash
gfrm resume
```

Or specify a session file explicitly:

```bash
gfrm resume --session-file ./sessions/my-session.json
```

### Migrate using explicit tokens (without settings)

Pass tokens directly via environment variables:

```bash
GFRM_SOURCE_TOKEN=glpat-xxx GFRM_TARGET_TOKEN=ghp-yyy gfrm migrate \
  --source-provider gitlab \
  --source-url "https://gitlab.com/group/project" \
  --target-provider github \
  --target-url "https://github.com/org/repo"
```

## `migrate`

Starts a migration from explicit source and target parameters.

Syntax:

```bash
gfrm migrate \
  --source-provider <github|gitlab|bitbucket> \
  --source-url <url> \
  --target-provider <github|gitlab|bitbucket> \
  --target-url <url> \
  [options]
```

Required:

- `--source-provider`
- `--source-url`
- `--target-provider`
- `--target-url`

Token sources (in order):

1. settings token (`token_env`, then `token_plain`)
2. env aliases (`GFRM_SOURCE_TOKEN`, `GFRM_TARGET_TOKEN`, provider aliases)

Main options:

- `--settings-profile <name>`
- `--skip-tags`
- `--from-tag <tag>`
- `--to-tag <tag>`
- `--workdir <dir>`
- `--log-file <path>`
- `--checkpoint-file <path>`
- `--tags-file <path>`
- `--download-workers <1..16>` (default: `4`)
- `--release-workers <1..8>` (default: `1`)
- `--dry-run`
- `--save-session` (default: enabled)
- `--no-save-session`
- `--session-file <path>`
- `--session-token-mode <env|plain>` (default: `env`)
- `--session-source-token-env <name>` (default: `GFRM_SOURCE_TOKEN`)
- `--session-target-token-env <name>` (default: `GFRM_TARGET_TOKEN`)
- `--no-banner`
- `--quiet`
- `--json`
- `--progress-bar`

Validation rules:

- `--download-workers` must be `1..16`
- `--release-workers` must be `1..8`
- if both `--from-tag` and `--to-tag` are set, `from <= to`

## `resume`

Resumes from a stored session file.

Syntax:

```bash
gfrm resume [options]
```

Main options:

- `--session-file <path>` (default when omitted: `./sessions/last-session.json`)
- `--settings-profile <name>`
- `--skip-tags`
- `--from-tag <tag>`
- `--to-tag <tag>`
- `--workdir <dir>`
- `--log-file <path>`
- `--checkpoint-file <path>`
- `--tags-file <path>`
- `--download-workers <1..16>`
- `--release-workers <1..8>`
- `--dry-run`
- `--save-session` (default: enabled)
- `--no-save-session`
- `--session-token-mode <env|plain>`
- `--session-source-token-env <name>`
- `--session-target-token-env <name>`
- `--no-banner`
- `--quiet`
- `--json`
- `--progress-bar`

Token sources (in order):

1. token from session context (`source_token` or `source_token_env`, same for target)
2. settings token (`token_env`, then `token_plain`)
3. env aliases (`GFRM_SOURCE_TOKEN`, `GFRM_TARGET_TOKEN`, provider aliases)

If a token is needed for a resume flow, set it via settings or env aliases before running `resume`.

## `demo`

Runs a local simulation flow with synthetic releases — no real source or target forge is required. Useful for verifying CLI behavior, artifact structure, and worker tuning without credentials.

Syntax:

```bash
gfrm demo [options]
```

Example:

```bash
gfrm demo --demo-releases 10 --demo-sleep-seconds 0.5
```

Main options:

- `--demo-releases <1..100>` (default: `5`) — number of synthetic releases to simulate
- `--demo-sleep-seconds <seconds>` (default: `1.0`, must be `>= 0`) — simulated processing delay per release
- `--skip-tags`
- `--dry-run`
- `--workdir`, `--log-file`, `--checkpoint-file`, `--tags-file`
- `--download-workers`, `--release-workers`
- `--session-file`, `--session-token-mode`, `--session-source-token-env`, `--session-target-token-env`
- `--no-banner`, `--quiet`, `--json`, `--progress-bar`

## `setup`

Interactive bootstrap for first-time settings configuration.

Syntax:

```bash
gfrm setup [options]
```

Options:

- `--profile <name>`
- `--local` (writes to `./.gfrm/settings.yaml`)
- `--yes` (non-interactive defaults)
- `--force` (run setup even when existing token mappings are detected)

## `settings`

Settings actions:

```bash
gfrm settings init [--profile <name>] [--local] [--yes]
gfrm settings set-token-env --provider <github|gitlab|bitbucket> --env-name <ENV_NAME> [--profile <name>] [--local]
gfrm settings set-token-plain --provider <github|gitlab|bitbucket> [--token <value>] [--profile <name>] [--local]
gfrm settings unset-token --provider <github|gitlab|bitbucket> [--profile <name>] [--local]
gfrm settings show [--profile <name>]
```

Behavior:

- global settings path: `~/.config/gfrm/settings.yaml` (or `XDG_CONFIG_HOME/gfrm/settings.yaml`)
- local override path: `./.gfrm/settings.yaml`
- effective settings = deep-merge(global, local)
- `settings show` masks `token_plain`
- `settings init` scans shell files in read-only mode (`.zshrc`, `.zprofile`, `.bashrc`, `.bash_profile`)

Profile resolution:

1. explicit `--settings-profile` (for `migrate`/`resume`)
2. `defaults.profile` from settings
3. `default`

## Runtime Invariants

- migration order is always tags-first
- release selection is semver-only (`vX.Y.Z`)
- checkpoint/idempotency and retry semantics are preserved
- run artifacts are always generated under a timestamped workdir

## Artifacts

Each run writes under:

```text
./migration-results/<YYYYMMDD-HHMMSS>/
```

Main files:

- `migration-log.jsonl` — newline-delimited JSON log; one event object per line
- `summary.json` — final run summary (schema v2)
- `failed-tags.txt` — one tag name per line; empty when all tags succeeded

`summary.json` fields:

- `schema_version`: `2`
- `command`: subcommand that was run (`migrate`, `resume`, `demo`)
- `status`: overall run outcome
- `retry_command`: pre-built `gfrm resume` command to retry failures (present only when `failed-tags.txt` is non-empty)

`failed-tags.txt` format:

```text
v1.2.0
v1.3.0
```

When this file is non-empty, `summary.json` will include a ready-to-run `retry_command`.

## Exit Codes

- `0`: successful run
- non-zero: validation or operational failure

## Common Issues

**Token not found**
`gfrm` exits non-zero with a validation error when no token is resolved for source or target. Run `gfrm settings show` to inspect current token mappings, or pass tokens via `GFRM_SOURCE_TOKEN` / `GFRM_TARGET_TOKEN`.

**Unsupported provider pair**
Same-provider migrations (`github -> github`, etc.) and Bitbucket Data Center are not supported. The CLI exits immediately with a validation error.

**`--from-tag` / `--to-tag` ordering**
If both are set, `from` must be `<=` `to` (semver order). Reversing them causes a validation failure before any migration starts.

**macOS Gatekeeper block**
If macOS blocks execution after unzip, run: `xattr -d com.apple.quarantine ./gfrm`

**Resume after session file loss**
If `./sessions/last-session.json` is missing, `gfrm resume` exits with an error. Start a new migration from scratch with `gfrm migrate` using the same parameters.
